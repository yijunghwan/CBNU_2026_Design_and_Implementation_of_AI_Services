3) LangGraph StateGraph 구조

노드 (12개)

securitycontext_loadplanner memoryslotstools responderjudgeretry_prep finalize_blockedfinalize_memoryfinalize_answer

조건부 분기 (conditional edges) 3곳

분기 노드조건이동
security민감정보/과다입력 감지차단 → finalize_blocked / 통과 → context_load
memory메모리 명령 처리 완료 여부처리됨 → finalize_memory / 통과 → slots
judge재시도 필요 & 상한(1회) 이내retry → retry_prep → tools / 그 외 → finalize_answer

반복 루프

judge → retry_prep → tools (최대 1회, 무한 루프 방지)

4) 단계별 내부 동작 & LLM 개입

단계(노드)역할LLM
security민감정보(주민번호/카드/전화/여권), 입력 길이 검사규칙 기반 (미들웨어)
context_load단기 대화창 + 장기 프로필/요약/최근대화 로드없음
planner의도 분석, 메모리명령 판단, 툴 계획, 슬롯 추출LLM 선택 모델
memory장기기억 ON/OFF·수정·삭제 FSM (동의/취소 확인)규칙 기반 (미들웨어)
slots지역/나이/소득/결혼/분야 표준값 정규화없음 (맵핑 테이블)
tools정책RAG·컨텐츠RAG·웹검색·파일파싱 오케스트레이션없음 (검색·평가)
responder툴 결과 + 맥락으로 최종 답변 초안 작성LLM 선택 모델
judge질문-근거-답변 정합성 평가, 재시도/되물음 결정LLM 기본 모델(settings)
사용자 선택 모델은 Planner/Responder에 적용되고, Judge는 기본 설정 모델로 실행됩니다.

보안 필터 주민등록번호·휴대폰·여권·신용카드 패턴 차단
입력 길이 제한 과도한 길이 요청 차단
메모리 확인 절차 장기기억 ON/OFF·수정·삭제 시 동의/취소 강제
예외 처리 툴 실패 시 개별 격리, 스트리밍 에러 이벤트 반환

6) Tool / RAG / Memory

Tool (4종, 오케스트레이션으로 복수 동시 실행 가능)

rag_policy_search rag_content_search web_search file_parse

RAG

  • 정책 / 컨텐츠 분리 벡터스토어 운영
  • 정책 검색: 질문 시 RAG로 1차 필터링하고, 하드조건(지역·나이·결혼·소득·분야) 불일치 감점 + 의미 유사도 가점으로 후보를 추림
  • 1차 추림 후 대화 내용 기반으로 정책명/내용과 응답을 LLM이 비교·평가해 최종 결과를 출력

Memory

  • 단기: 세션 대화창 (in-memory)
  • 장기: 유저코드 기반 SQLite (프로필 + 요약 + 최근대화)

7) 상세 사용 가이드

기본 사용 순서

1
서버 실행: python -m agent3.run_server (포트 8002)
2
브라우저 접속: http://localhost:8002
3
상단에서 모델 선택 → 질문 입력 → 전송
4
필요 시 📎로 파일 첨부 후 파일 기반 질문
5
우측 패널에서 계획·슬롯·툴·메모리 상태 실시간 확인

상단 선택 항목 안내 (유저코드 / 모델)

  • 유저코드 입력창: 같은 유저코드를 쓰면 같은 장기기억이 연결되고, 다른 코드는 독립 메모리로 분리됩니다.
  • 모델 선택 드롭다운: 선택 모델은 Planner/Responder에 적용되고, Judge는 기본 설정 모델(.env의 LLM_PROVIDER/LLM_MODEL_NAME)을 사용합니다.

장기기억 사용법

  1. 상단 유저코드 입력 (예: user_kim_01)
  2. "장기기억 켜줘" 또는 "내 프로필 저장할게"로 요청
  3. 시스템이 확인을 물으면 "동의" 또는 "확인"으로 확정 (취소하려면 "취소")
  4. 나이·지역·소득·결혼여부·관심분야를 자연어로 알려 프로필 갱신
  5. "내 저장 프로필 보여줘"로 조회, "장기기억 꺼줘"로 비활성화

장기기억에서 개인정보 변경/삭제하는 법

  • 개인정보 변경: "개인정보 변경: 만 26세, 서울, 취업준비중, 미혼"처럼 요청
  • 시스템 확인 질문이 오면 동의 또는 확인으로 확정 (취소하려면 취소)
  • 개인정보만 삭제: "개인정보 삭제" 요청 후 확인 단계에서 동의
  • 결과: 프로필 슬롯(나이/지역/소득/결혼/주거/고용 등)은 삭제되고, 장기 대화기록은 유지

장기기억에서 대화내용 삭제하는 법

  • 장기 대화기록만 삭제: "대화기록 삭제" 요청 후 확인 단계에서 동의
  • 결과: 장기 저장 대화만 삭제되며, 프로필 정보는 유지
  • 전체 삭제: "장기기억 전체 삭제" 요청 후 동의하면 개인정보+대화기록 모두 삭제
  • 주의: 전체 삭제 후에도 장기기억 모드는 켜진 상태이므로, 원하면 "장기기억 꺼줘"를 추가로 실행

확인/취소 제어 규칙

  • 변경/삭제 계열 요청은 즉시 실행되지 않고 확인 단계가 먼저 실행됩니다.
  • 확정 단어: 동의, 확인, 네, 예, 진행
  • 취소 단어: 취소, 중단, 아니요, 보류

유저코드 규칙

  • 유저코드는 장기기억 식별자입니다. 같은 코드 = 같은 프로필·이력 연결
  • 다른 코드를 쓰면 완전히 독립된 메모리 공간
  • 유저코드를 비우면 장기기억 없이 단기 대화만 유지

프로필 저장 방식

  • 장기기억이 ON일 때만 프로필/대화가 SQLite에 저장됩니다
  • 슬롯(지역·나이·소득·결혼·주거·분야) 단위로 표준화되어 보관
  • 다음 대화에서 자동으로 컨텍스트에 주입되어 맞춤 추천에 반영

모델 선택 가이드

  • 정확한 구조화·추론 중심: 고성능 모델
  • 빠른 응답·저비용 중심: 경량 모델
  • 선택 모델은 Planner/Responder에 반영되고, Judge는 기본 설정 모델을 사용

질문 방식 & 추천

  • 맞춤 추천: "서울 거주, 27세, 미혼, 중위소득 100%, 취업 분야 정책 추천"
  • 컨텐츠 탐색: "청년 대상 모집 공고나 최신 소식 알려줘"
  • 최신성 강화: "최근 마감 임박 정책 위주로 알려줘"
  • 파일 기반: 파일 첨부 후 "이 파일 조건에 맞는 정책만 추천해줘"

9) 한계점 및 개선 방향

한계점

  • 장기기억 요약이 현재 규칙 기반이라 LLM 요약 대비 품질이 제한적입니다.
  • RAG 정량 평가셋(Precision@K 등)이 아직 문서화되지 않았습니다.
  • 웹 검색은 키(TAVILY) 설정 시에만 동작합니다.
  • 데이터(정책/컨텐츠 원천 및 벡터 인덱스)가 동적으로 변경될 때 자동 감지/재색인 동기화 장치가 없어 최신 데이터 반영이 지연될 수 있습니다.
  • 한 턴에서 LLM을 계획(Planner) + 답변생성(Responder) + 평가(Judge)로 평균 약 3회 호출합니다(질문 유형에 따라 security/memory 단계에서 조기 종료될 수 있음).
  • Judge 루프가 1회 발생하면 Responder/Judge가 추가 호출되어 최대 5회 LLM 호출이 일어날 수 있습니다.
  • 위 구조로 인해 질문 난이도/루프 여부에 따라 토큰 사용량과 비용 편차가 커질 수 있습니다.
  • 청년 정책 도메인을 벗어난 일반 대화로 갈수록 응답 품질과 근거 정합성이 눈에 띄게 낮아질 수 있습니다.
  • 현재 운영 기준으로 실제 평가(Judge)는 .env의 LLM_PROVIDER/LLM_MODEL_NAME에 명시된 경량 모델(예: gpt-4o-mini)을 사용합니다.

개선 방향

  • 평가(Judge)는 고성능 생성 모델과 분리해 경량 모델을 별도로 명시해 운용(비용 최적화).
  • 평가 단계 전용 provider/model 설정 키를 분리해 운영자가 비용/품질을 독립적으로 튜닝.
  • 청년정책 외 대화 의도를 조기 감지하는 도메인 분류/라우팅을 추가해, 비정책 질문은 일반형 답변 모드로 전환.
  • 비정책 질의 포함 품질 평가셋을 별도로 구축해(정확성/근거성/일관성) 품질 저하 구간을 지속적으로 보정.
